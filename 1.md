
# 整体定位
**V0.1：基础可用（贪心就近重平衡 + 直线距离 + 固定20%）**
**V0.2：学术标准（最小成本流 MCF + 真实路网时间 + 多目标优化）**


---

# 一、0 主循环引擎（Main Engine）优化
## 当前问题
- 重平衡是**固定周期触发**（如每30s），不看拥堵强度
- 行驶时间**不实时更新**，调度决策滞后
- 车辆状态更新与 SUMO 仿真步不同步

## 优化方向
1. **自适应重平衡触发**
   拥堵指数 > 阈值 才触发重平衡（更智能）
2. **统一时间同步机制**
   所有成本、派单、重平衡**共用同一套 travel_time**
3. 增加 **路网状态快照（traffic_state）**
   每步保存：速度、拥堵、红绿灯、路段耗时


---

# 二、1 空间索引模块（Index Design）优化
## 当前问题
- 只有 R‑Tree 存 (x,y)
- **没有区域(Cell)索引**
- 没有**区域中心边、区域代表点**
- 无法构建成本矩阵

## 必须优化（极小改动，巨大提升）
1. 增加 **Cell‑ID 索引**
   每个网格存：中心坐标、中心边、平均速度、拥堵等级
2. 构建 **cell_to_edge 字典**
   `cell_center_edge[i][j] = edgeID`
3. 构建 **区域邻接表**
   只计算相邻区域成本，避免全矩阵 O(n²)

## 作用
给**最小成本流提供网格节点**。

---

# 三、2 订单处理模块（Request Handling）优化
## 当前问题
- KNN 只看**直线距离**
- 不考虑**拥堵、接驾时间、到达可行性**

## 优化（关键！真实度飞跃）
1. **KNN 改为 Time‑KNN**
   不是最近车，是**最快到达车**
2. 接驾时间 = 真实行驶时间（含拥堵+红绿灯）
3. 过滤：超过 max_wait_time（如 300s）不派单

## 效果
订单完成率更真实，不会出现虚假快速匹配。


---

# 四、3 派单模块（Dispatch Logic）优化
## 当前问题
- 贪心派单
- 无全局最优
- 无多目标：最快、最近、最少空驶、最高完成率

## 优化（多目标派单）
多目标打分机制（权重可调）：
```
score = w1 * 到达时间 + w2 * 车辆空驶时长 + w3 * 区域未来需求
```
选择**综合 score 最低**的车。

## 作用
保证：
- 乘客等待最少
- 车辆空驶最少
- 全局效率最高


---

# 五、4 车辆状态模块（Availability）优化
## 当前问题
- 只有 IDLE/PICKUP/OCCUPIED
- 没有 **REBALANCING** 状态
- 无法防止：重平衡车被派单 → 冲突

## 优化
1. 新增状态：**REBALANCING**
2. 重平衡中的车**不参与派单**
3. 到达目标区域 → 自动回到 IDLE
4. 数据库原子状态切换，避免冲突

这是**系统稳定的关键**。

---

# 六、5 重平衡模块（Rebalancing）—— 核心升级
## 当前（V0.1）
贪心、就近、每个区域最多20%、直线距离、简单供需。

## V0.2 你要的三大核心：
### 1. 真实行驶时间（非直线、非距离，含拥堵+红绿灯）
```
travel_time(i→j) = 路段长度 / 实时速度
                 + 红绿灯等待
                 + 拥堵惩罚(congestion_factor)
```
完全由 SUMO 实时计算。

### 2. 成本矩阵 CostMatrix(i,j) = 时间
构建 **Grid × Grid 时间矩阵**，作为最小成本流输入。

### 3. 最小成本流（学术标准 Rebalance）
问题建模：
- **Source**：盈余区域（可出车）
- **Sink**：赤字区域（需要车）
- **Cost**：i→j 行驶时间
- **Flow**：车辆数
- **Constraint**：每个区域最多流出 20%
- **Objective**：min total time → 最少空驶
- **Objective**：max fulfilled demand → 最大完成率
