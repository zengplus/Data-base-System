\begin{abstract}
本文提出一种基于R-Tree空间索引的自动驾驶出租车时空调度系统，旨在解决大规模车队下供需时空错配与实时派单瓶颈。系统集成K近邻（KNN）查询实现毫秒级车辆匹配，并引入动态再平衡模块主动调节空间供需缺口。在覆盖3km$\times$3km都市网格、含3014条有效边及1500个高峰时段请求的SUMO仿真中，对比三种调度策略（随机基线、纯KNN、完整系统）与三种车队规模（100/200/300辆）。结果表明：当车队规模增至300辆时，本文系统派单成功率达62.0\%，平均等待时间低至195.1秒，与纯KNN性能相当；同时通过再平衡将车队空闲率从83.7\%大幅压降至31.3\%，显著提升全局资源利用率。
\end{abstract}

\begin{IEEEkeywords}
自动驾驶出租车，时空间索引，动态再平衡，K近邻查询
\end{IEEEkeywords}

\section{引言}
高峰时段下，自动驾驶出租车调度面临每秒数十次请求与数百辆车动态位置更新的双重压力。传统派单算法每匹配一次需遍历全部空闲车辆，时间复杂度$O(N)$，严重制约系统实时性与可扩展性。

本文引入R-Tree空间索引，将K近邻查询的期望时间复杂度降至$O(\log N)$，并基于此构建实时调度架构。主要贡献包括：
\begin{enumerate}
    \item 提出紧耦合R-Tree与KNN派单的调度系统，利用SQLite R-Tree模块与原子事务保障状态一致性。
    \item 设计基于网格供需缺口的动态再平衡策略，主动引导冗余车辆至高需求区域，缓解纯贪心派单造成的空间资源浪费。
    \item 在SUMO大规模路网（3014条有效边，3km$\times$3km）中，评估车队规模三倍增长下系统性能，验证索引结构与调度策略的可扩展性。
\end{enumerate}

\section{系统模型与问题描述}

\subsection{系统场景定义}
本文研究场景基于美国纽约部分真实路网截取的都市网格（路网覆盖约 3km $\times$ 3km 区域，包含3014条有效有向边，满足$>$500边要求），利用OSM和SUMO Netconvert工具生成。经过清洗与有效边提取，该路网涵盖主干道、次干道和支路（排除内部虚边）。路网中部署有2000辆背景车辆以模拟真实交通流背景。为模拟早晚高峰的潮汐车流特征，背景车辆内置了带自定义启发式的A*路径规划算法，严厉惩罚穿越市中心核心区域的路径，从而制造中心地带拥堵、外围相对畅通的逼真交通态势。此外，路网中还部署了一支规模在100至300辆之间可变的自动驾驶出租车车队。在3600秒（1小时）的高峰仿真时段内，系统需处理1500个随机生成的乘客行程请求。为构造极端的空间不平衡场景，80\%的请求基于固定随机种子被强制约束在路网中心热点区域。

调度系统的核心目标是在线地为每个乘客请求实时匹配最合适的空闲出租车，并通过SUMO的TraCI接口控制车辆执行接送任务，最终达成最小化平均乘客等待时间（Average Waiting Time, AWT）和最大化订单完成率的双重优化目标。

\subsection{出租车状态机与原子性约束}
每辆出租车在任意时刻处于以下四种互斥状态之一：
\begin{itemize}
    \item \textbf{IDLE（空闲）}：车辆未被分配任务，可自由巡游或驻留，是派单候选集合的唯一来源。
    \item \textbf{PICKUP（接驾）}：车辆已被分配请求，正在驶向乘客出发点的途中。
    \item \textbf{OCCUPIED（载客）}：车辆已接到乘客，正在驶向目的地的途中。
    \item \textbf{REBALANCING（再平衡中）}：车辆正处于被系统主动调度、驶向高需求缺车区的途中。
\end{itemize}
状态转换遵循严格的流转逻辑。状态管理必须满足原子性约束——任何时刻一辆出租车不能被两个请求同时分配。本文通过SQLite数据库的显式事务机制保证状态更新的隔离性与原子性。

\subsection{性能评估指标}
为全面评估调度系统的服务质量与运营效率，本文采用以下核心指标：
\begin{itemize}
    \item \textbf{订单完成率}：成功完成匹配的请求数占总请求数（1500个）的百分比，是衡量服务能力的核心指标。
    \item \textbf{平均乘客等待时间（AWT）}：请求生成时刻至出租车到达出发边时刻的平均时长，包含排队等待时间，是衡量乘客体验的首要指标。
    \item \textbf{空车率（Idle Rate）}：仿真时段内处于空闲状态的车辆数占车队总数的平均比率，反映运力闲置程度。
    \item \textbf{空驶率（Empty-Mile Ratio）}：车辆无客行驶里程（含再平衡调度与接驾行程）在总行驶里程中的占比，衡量为达成供需平衡所付出的额外行驶成本。
\end{itemize}

\section{方法论}

本文提出的自动驾驶出租车时空调度系统由多个核心模块协同构成。Python控制器作为中枢，通过TraCI接口与SUMO仿真环境双向交互，同时将车辆状态与空间位置持久化于SQLite数据库中。

\subsection{基于R-Tree的空间索引设计}
为支撑高并发的近邻车辆检索，我们在SQLite数据库中构建了R-Tree虚拟表以存储车辆的实时二维坐标。数据库表结构设计如下：

\begin{verbatim}
-- 出租车主表
CREATE TABLE taxis (
    taxi_id TEXT PRIMARY KEY,
    status TEXT DEFAULT 'IDLE',
    current_x REAL,
    current_y REAL,
    cell_id TEXT
);

-- R-Tree空间索引虚拟表
CREATE VIRTUAL TABLE taxi_locations USING rtree(
    id INTEGER,        -- 关联 taxis.taxi_id
    minX REAL, maxX REAL,
    minY REAL, maxY REAL
);
\end{verbatim}

每当车辆在SUMO中移动时，控制器通过TraCI获取其新坐标$(x,y)$，并以原子操作方式增量更新R-Tree索引。R-Tree将空间递归划分为层次化的最小边界矩形（MBR），使得KNN查询可通过剪枝大量无关节点，将期望时间复杂度从全表扫描的$O(N)$降至$O(\log N)$。

\subsection{K近邻派单与路径规划}
当新请求生成时，系统立即执行以下KNN查询，获取距离请求点欧氏距离最近的15辆空闲出租车：

\begin{verbatim}
SELECT t.taxi_id
FROM taxi_locations r JOIN taxis t 
ON r.id = CAST(SUBSTR(t.taxi_id, 6) AS INTEGER)
WHERE t.status = 'IDLE'
ORDER BY (minX-?)*(minX-?) + (minY-?)*(minY-?) ASC 
LIMIT 15;
\end{verbatim}

在获取候选车辆集后，控制器调用TraCI的\texttt{getShortestPath}接口，计算各候选车辆到乘客出发点的真实路网行驶时间，选取预估接驾时间最短者作为派单目标。随后为目标车辆设置导航路径，并将其状态更新为PICKUP。

\subsection{原子事务状态管理}
为防止并发派单导致的状态错乱，我们采用SQLite事务对状态更新进行加锁保护。核心派单逻辑封装在以下原子操作中：

\begin{verbatim}
BEGIN TRANSACTION;
UPDATE taxis SET status='PICKUP' WHERE taxi_id=?;
UPDATE trip_requests SET status='ASSIGNED' 
  WHERE request_id=?;
INSERT INTO assignments (request_id, taxi_id, ...) 
  VALUES (?, ?, ...);
COMMIT;
\end{verbatim}

\subsection{动态再平衡策略}
纯KNN派单属于被动响应模式，其贪心本质在请求密度空间分布不均时，容易导致局部车辆冗余与局部车辆匮乏并存。为此，我们设计了动态再平衡模块，以30秒为周期主动调度冗余车辆。

\subsubsection{供需缺口量化}
将路网划分为均匀网格单元。对每个单元$c$，统计最近时间窗内的请求数量$D(c)$，并实时统计当前处于IDLE状态的车辆数$S(c)$。定义供需缺口为$D(c) - S(c)$。当该值大于触发阈值$\theta$（本文取$\theta=5$）时，该单元被标记为再平衡目标区域（缺车区）；反之则为源区域（富余区）。

\subsubsection{调度执行}
算法筛选供需缺口最大的目标单元，并从源区域的空闲车辆中调拨不超过其总量20\%的车辆，通过TraCI下达前往高需求区域的指令。被调拨车辆的状态临时标记为\texttt{REBALANCING}，到达目的地后自动恢复为IDLE，从而实现动态的供需空间均衡。

\section{实验设置}

\subsection{仿真平台与参数}
实验基于SUMO 1.20.0与Python控制器。为保证实验结果的完全可复现性，我们通过设置统一的随机种子（\texttt{seed=42}），固定了背景车辆的初始位置以及1500个乘客请求的时空分布。乘客请求的超时等待阈值设定为600秒，超出该时限未获接驾的请求标记为失败。

\subsection{对比策略与变量设计}
为全面评估本文系统（记为\textbf{Proposed}）的性能优势，设置两组对照策略：
\begin{itemize}
    \item \textbf{Baseline（随机派单）}：从空闲车辆中随机选取一辆派单，无智能路由与再平衡机制。
    \item \textbf{KNN Scheduling（纯KNN派单）}：采用R-Tree加速的KNN匹配，但关闭动态再平衡模块。
\end{itemize}
实验采用双变量设计：调度策略（3种）$\times$ 车队规模（100、200、300辆），共计9组实验，以全面评估车队密度提升三倍时各策略的响应特征与扩展能力。

\textbf{结果分析：}
\begin{itemize}
    \item \textbf{派单成功率与平均等待时间——“贪心陷阱”现象}：
    
    在高峰时段仿真中，80\%的订单被强制集中在市中心热点区域。纯KNN策略凭借其“就近接单”的贪心本质，使车辆在完成热点订单后直接停留原地并迅速接到下一单，因而在短时间内的局部响应极快（完成率64.0\%，等待时间191.6秒）。

    相比之下，Proposed系统（完成率62.0\%，等待时间195.1秒）为了追求全局供需均衡，主动将外围空闲车辆跨区调度至缺车区域。由于车辆在REBALANCING状态下无法接单，这一过程在短时统计中引入了额外的时间损耗，导致其完成率与等待时间略逊于纯KNN，但两者基本处于同一水平。

    \item \textbf{空车率与空驶率的权衡——核心优化分析}：
    
    尽管在接单效率上两种策略表现相近，但在车队资源利用率方面，Proposed系统展现出压倒性优势。如图\ref{fig:idle}所示，在300辆车规模下，由于请求分布的极端不均衡，纯KNN策略导致大量完成外围订单的车辆永久滞留于低需求区域，空车率高达83.7\%；尽管其空驶率极低（仅15.1\%，因车辆完单后几乎不移动），但这实质上构成了巨大的运力浪费。

    而Proposed系统通过动态再平衡模块，主动将闲置车辆引导回市中心热点区域，成功将空车率大幅压降至31.3\%。作为代价，如图\ref{fig:empty_mile}所示，Proposed策略的主动调拨使其空驶率上升至38.6\%。这种“以可控的再平衡空驶换取全局供需匹配”的策略，有效消除了车辆的无效停滞，实现了车队运力在空间维度上的全局深度激活。
\end{itemize}

\subsection{空间索引深度与派单性能的关联分析}
实验结果显示，随着车队数量翻倍，KNN匹配过程的平均耗时并未呈现线性增长，而是保持了常数级的极低延迟。这一优良的对数级时间复杂度特性，保证了系统即使在300辆车并发运行的场景下，仍能为每个请求实时、准确地定位最近的空闲车辆集合。

正由于计算瓶颈被有效突破，系统派单成功率并未因规模增加而导致的潜在拥塞出现下降，反而随运力供给的日益充沛，从100辆时的49.3\%稳步提升至300辆时的62.0\%。这充分证明，空间索引深度的缓慢增长对系统实时性的影响完全可控，为大规模自动驾驶车队的规模化派单提供了有力支撑。

\section{结论}

本文设计并实现了一个面向大规模自动驾驶出租车车队的时空调度系统。系统以SQLite R-Tree为核心构建空间索引，利用KNN匹配实现毫秒级快速派单，并通过原子事务保证状态流转的一致性。针对纯贪心派单在极端空间失衡下引发的车辆闲置问题，引入了基于供需缺口感知的动态再平衡策略，主动引导冗余运力向高需求区域集中。

在SUMO仿真平台上，基于1500个高峰时段乘客请求与三种车队规模的对照实验，本文验证了系统的有效性与可扩展性。当车队规模增至300辆时，所提系统在保持订单完成率（62.0\%）和平均等待时间（195.1秒）与纯KNN策略基本相当的前提下，成功将车队空闲率从83.7\%断崖式降低至31.3\%，同时空驶率控制在38.6\%。

上述结果表明，本文系统成功解决了核心的“贪心陷阱”问题：通过接受一定程度的再平衡空驶代价，换取了供需空间分布的高度匹配与车队全局资源的高效盘活。这一权衡揭示了大规模按需出行系统中空间效率与行驶成本之间的深层关联，为同类时空调度系统的设计提供了有价值的参考。
